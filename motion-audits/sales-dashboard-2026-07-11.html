This is a disciplined codebase — custom easing tokens, a global reduced-motion guard, and two genuinely well-built FLIP modals. CriticalEmil

The KPI loading skeleton is a hard cut, in both directions

What#kpi-skeleton toggles display:none ↔ block with zero transition — no fade, no delay easing. It fires on every data load: first load, session resume, and any manual refresh. index.html:3552-3553, toggled at index.html:5602,5612,5622,5626.

Why it mattersThis is the single most-repeated state change in the app, and it's the one place the file's entire easing system doesn't reach. A restrained tool still needs to feel intentional at its most frequent moment — right now it feels like an unfinished render, not a designed wait state.

Recommended motionA fast 180ms opacity + 4px translateY cross-fade between skeleton and content, using the existing var(--ease-out) token — no new easing curve needed, and still comfortably under Emil's 300ms ceiling for a frequent, non-keyboard action.

The date-range popover scales from its own center, not its trigger

What#drp-popup animates scale-in (0.94→1, 200ms) with no transform-origin set, so it defaults to dead center rather than the date-picker button that opened it. index.html:2501-2508, keyframe at index.html:312-315.

Why it mattersThe popup is already positioned near its trigger, but the un-anchored scale breaks the spatial thread between "what I clicked" and "what appeared." The effect is small at a 6% scale delta — this is exactly the kind of free detail Emil calls out as cheap to fix and easy to skip.

Recommended motionSet transform-origin inline from the trigger button's screen position before the scale-in plays — the same coordinate math the KPI modal's own FLIP logic already computes elsewhere in this file, just reused for a simpler scale instead of a full morph.

Jakub Krehel — Production Polish

Secondary lens

Concern

What's working well
  • ✓The KPI modal and channel popup both FLIP-morph from the exact card that was clicked — position, size, and border-radius all interpolate from the source element. This is real, hard-won craft. index.html:9147-9157, index.html:8717-8724
  • ✓Both modals' close animation uses a distinct, slightly faster curve (cubic-bezier(0.5,0,0.75,0), 420ms) than their open (--ease-smooth, 620ms) — enter and exit are not identical copies of each other at the modal level.
  • ✓Chart gradients are cached per chart instance (chart._gradCache) instead of recreated every animation frame — a real performance detail, not just a visual one.
Issues to address
ImportantJakub

The toast's exit is a literal mirror of its enter

What.toast and .toast.show use the identical opacity + translateY(8px) transition, 220ms, for both directions — the exit is the enter played backward, then the node is removed 240ms later. index.html:3148-3152, timed at index.html:5591.

Why it mattersJakub's rule here is specific: exits should read as subtler than enters, not as a reverse of the same motion. A mirrored toast isn't broken — most apps get this wrong entirely by having no exit transition at all — but it's the one place where "finished" and "considered" still diverge in this file.

Recommended motionKeep the enter as-is. On exit, halve the distance (8px → 3px) and shorten slightly (220ms → 180ms) so it reads as a settle rather than a rewind — same easing token, no new curve.
